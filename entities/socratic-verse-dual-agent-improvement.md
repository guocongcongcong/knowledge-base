# 双 Agent 交叉验证：一次 Socratic Verse 系统的全面改进实录

> 2026-06-22 | 作者：锅釨

---

## 一、背景

**Socratic Verse** 是一个零代码的 AI 家教系统——把教材放进 `materials/` 目录，AI 教师用苏格拉底追问法带你学习。支持 8 种主动学习方法（V1-V8），3 位 AI 教师轮值（三月七、甘雨、刻晴），全部由 markdown 文件驱动，无数据库、无后端。

今天（2026-06-22）之前，这个系统已经运行了一周。锅釨用 V1 苏格拉底模式和 V7 对子互教模式学完了入门篇核心链 5 篇教材（理解 Agent、写 Skill、工具与 MCP、上下文工程、驾驭工程），上完了 6 节课。从表面看，所有教材都已标记 ✅，一切正常。

直到今天下午，我们用 **Hermes Agent（刻晴角色）** 和 **Claude Code** 做了两轮独立分析——两份分析交叉验证后，发现了 5 个严重问题，推动了一次涉及 7 个文件、8 项 P0 改进的系统级重构。

---

## 二、问题发现：6 节课背后的"假象"

锅釨上完 6 节课后，总觉得"进度条走了但有点空虚"——教材全标记 ✅ 了，但知识没有形成体系感。直觉告诉他，系统的进步映射出了问题。于是我们做了两件事：

1. **关联图谱分析**（Hermes 分析所有教材的依赖关系）
2. **V1-V8 体系评估**（双 Agent 独立分析）

结果发现 5 大问题：

### 🔴 问题 1：扫读 ≠ 学习

关联图谱分析时通读了所有教材全文，系统误以为"读过 = 学过"，直接把教材标记为 ✅。学习者还没被追问，教学就已经被跳过了。这是系统架构级别的语义概念混淆——分析模式和学习模式共享同一个进度标记。

### 🔴 问题 2：V2-V8 大面积未生效

8 种方法的宣传，实际执行过的方法只有 **2 种**——V1 苏格拉底（5 次）和 V7 对子互教（1 次）。其余 6 种方法（V2 费曼、V3 Anki、V4 骰子、V5 合书复述、V6 交错魔鬼、V8 自适应）定义在 `prompts/system-prompts.md` 中，但在 6 节课中从未被触发过。根因有三：

- 没有自动触发机制——用户必须手动指定版本号
- V3/V4 依赖的题库从未建立
- 没有学习方法调度表定义每次课后的复习链

### 🔴 问题 3：老师轮值冲突

旧 `system_detail.md` 的轮值规则说"顺序是三月七→甘雨→刻晴"，但实际问题更复杂：三月七和甘雨都不适合教系统架构类的"驾驭工程"，而刻晴是唯一的架构老师。结果"驾驭工程"先由三月七硬教了一轮，又由刻晴重讲一遍——浪费了一节课。

更致命的是，轮值规则中只说了"顺序"，没说"学科偏好不匹配时怎么办"。

### 🟡 问题 4：教材 order 编号错位

旧 `materials-map.md` 已标记了这个问题（P0 优先级），但一直未执行。三篇权威参考文献（Anthropic、OpenAI、LilianWeng）的 order 编号是 1-3，排在所有核心教材之前。结果按 order 学习时，学习者先读了三篇参考文献，再读核心教材——完全违反认知规律。参考文献应该在理解核心概念后作为"反刍"材料，而不是前置挡路石。

### 🟡 问题 5：复习机制完全缺失

这是最严重的一个。核心链 5 篇全部标记 ✅，但没有任何后续复习计划。2026-06-16 学的"理解Agent"到 2026-06-22 已经过了 6 天——按艾宾浩斯遗忘曲线，已遗忘约 50%。`progress.md` 只有"学了什么"，没有"什么时候该复习什么"。

## 三、双 Agent 交叉验证

发现问题的过程是这样的：

**第一轮：Hermes Agent（刻晴角色）** 读取所有教材和教师文件，生成了一份完整的关联图谱（`materials-map.md`），标出了概念依赖关系、矛盾、重复，给出了 P0-P2 改动优先级。

**第二轮：Claude Code** 独立读取相同的文件和教材，做了 V1-V8 体系评估。它不知道 Hermes 已经写了什么。

然后两份分析合并对比：

### 双方共识（占比 ~80%）

| 问题 | 严重度 |
|------|:------:|
| V2-V8 大面积未生效 | 🔴 |
| 复习机制完全缺失 | 🔴 |
| 教师轮值混乱 | 🔴 |
| order 编号错位 | 🟡 |
| 扫读 vs 学习混淆 | 🟡 |

### Claude Code 独有发现（3 项）

1. **V2 费曼是 V1 苏格拉底的"镜像"** — 苏格拉底是老师追问学生，费曼是学生教老师。两者结合才能同时覆盖"被追问"和"主动讲"两个方向。之前只看重前者，忽略了后者的互补价值。
2. **V7 的实际效果有数据支撑** — 2026-06-16 的 V7 课中，学习者主动创造了"削铅笔"类比来解释 Agent Loop 的三层出口——这是"对子互教"模式产生"主动提取"效果的直接证据，不只是理论假设。
3. **V8 不应急于启用** — 在 V1-V7 各自的效果还没验证完之前，V8 的调度决策没有依据。V8 是"管家"，不是"默认模式"。

### Hermes 独有发现

- 关联图谱中识别了 3 处跨教材概念矛盾和 4 处概念重复
- 给出了精确的 P0-P2 优先级分级
- 发现"驾驭工程"的轮值冲突有结构性原因（架构类教材只有刻晴擅长）

## 四、决策原则：8 项 P0 改进谁方案

两份分析合并后，我们列出了一个包含 15 项改进的清单，按 P0/P1/P2 分级。P0 是"本周必须完成"，共 8 项。每项的最佳方案并不总是来自同一份分析：

### 全部 8 项 P0 改进

| # | 改进 | 方案来源 | 为什么选这个方案 |
|---|------|---------|----------------|
| 1 | V5 合书复述 cron 今晚执行 | **Hermes**（作为执行 Agent） | 执行层面的决策，谁跑都一样。Cron 的时机（课后 1h 内）由学习科学决定，不是偏好 |
| 2 | 每篇核心链设定 V2 cron | **Hermes**（作为执行 Agent） | 同上。V5→V2→V7→V4 的复习链顺序是学习科学，不是分析偏好 |
| 3 | 生成 Anki 推理题（5 篇 25 道） | **Claude Code**（题库内容） | 这个分工最自然：Claude Code 擅长生成结构化内容和场景题。25 道题全部用"场景推理"而非"定义回忆"，每道有详细参考答案 |
| 4 | 设定每日 V3 cron（08:30） | **Hermes**（执行层） | Cron 调度是 Hermes 的 native 能力，Claude Code 没有 cron 系统 |
| 5 | 升级 progress.md 复习跟踪字段 | **双方共识** | 字段设计来自 V1-V8 体系评估中的需求——"最后复习/下次复习/复习次数/Anki"四字段几乎是双方分析交集的直接产物 |
| 6 | 区分学习模式 vs 分析模式 | **Hermes**（架构决策） | Hermes 作为系统设计 Agent，更清楚如何通过语义识别规则（包含"分析/扫读/对比"等关键词 → 分析模式；"教我/学/上课"→ 学习模式）来区分两种模式 |
| 7 | 修正教材 order → core_chain | **Hermes**（materials-map 自带） | 这个方案已经存在于 Hermes 生成的关联图谱中，直接执行即可 |
| 8 | 修复教师轮值逻辑 | **Claude Code**（细节补充） | Claude Code 补充了学科偏好冲突解决的细节——偏差 ≤1 个位置按偏好；≥2 个位置按轮值并记录备注。Hermes 的关联图谱补充了"驾驭工程固定由刻晴教授"的例外规则 |

### 决策原则

回顾整个决策过程，实际上遵循了三层原则：

1. **谁离数据最近谁出方案** — 题库设计、轮值细节这类需要具体场景洞察的，Claude Code 更擅长。架构设计、优先级分级这类需要系统思维的，Hermes 更擅长。
2. **执行层谁有工具谁动手** — Cron 调度、文件写入这类操作，由拥有 cron 系统和写文件权限的 Agent 执行。两个 Agent 各有所长，不需要强求同一个 Agent 做所有事。
3. **有共识的部分直接执行** — progress.md 的复习字段、学习模式 vs 分析模式的区分这类双方分析高度一致的内容，不需要争论"谁的方案更好"，直接执行最快。

## 六、反思：双 Agent 协作模式的价值

这次经历最核心的收获不是"修了多少 bug"——而是我们验证了一套可复用的双 Agent 协作模式。

### 什么时候该并行（分开分析，各自独立）

- **系统审计类任务** — 评估系统健康状况、发现隐藏问题。两个 Agent 的盲区不同，Hermes 更容易发现系统架构级别的裂缝（"扫读 vs 学习"混淆根在于没有区分两种模式），Claude Code 更容易发现细节层面的缺失（"V2 是 V1 的镜像"需要理解具体教学法的互补关系）。
- **数据分析和洞察生成** — 两份独立分析交叉验证后，共识部分可以无争议执行，差异部分反而产生了新的洞察（如"V8 不应急于启用"这个结论来自两份分析的共识空白，而不是共识）。
- **内容生成** — 题库、骰子题这类结构化内容，谁都可以独立完成。并行生成后合并，效率翻倍。

### 什么时候该串行（A 产出 → B 评审 → 合并执行）

- **决策链中有依赖关系时** — 先由 Hermes 做关联图谱分析（输出材料依赖关系），再由 Claude Code 基于图谱做 V1-V8 体系评估。如果直接让 Claude Code 单独做体系评估，它需要花时间自己梳理教材关联关系。
- **需要协商优先级时** — P0/P1/P2 的分级需要双方数据都有。先各自分析，再合并排序，比让一个 Agent 独自做优先级决策更客观。

### 这次协作的意外收获

**Claude Code 发现但被 Hermes 的关联图谱验证的"V8 不应急于启用"结论**是一个很好的例子：两份分析不是简单的"共识+差异"二分——还有第三种情况："A 没注意到 B 的某条结论，但 A 的数据恰好支持了 B 的结论"。这种 1+1>2 的效果，是双 Agent 协作超越单 Agent 的核心价值。

### 一套可复用的协作框架

```
1. 分工定义 ──→ 谁分析什么、谁产出什么
2. 并行执行 ──→ 双方独立完成，不交互
3. 合并对比 ──→ 共识 → 直接执行
               差异 → 评估谁的数据更 direct
               空白 → 是否是新洞察
4. 方案选择 ──→ 谁离数据最近谁出方案
               谁有工具谁动手
               有共识直接执行
5. 执行验证 ──→ 所有改动落实后，再交叉验证一次
```

## 尾声

截止今晚 22:00，Socratic Verse 的首次完整学习生命周期测试将启动：V5 合书复述 → V2 费曼 → V7 对子互教 → V4 骰子，串行执行 4 种方法覆盖 5 篇核心教材的全部内容。

这次改进不是"修 bug"——它实际上是让 Socratic Verse 从"单方法学习系统"升级为"多方法生命周期系统"。V1-V8 不再是一个可选菜单，而是一个完整的学习引擎：V1 是种子，V5 是收割，V2 是检验，V7 是巩固，V3 是保鲜，V4 是惊喜，V6 是体检，V8 是管家。

双 Agent 做的事，任何一个 Agent 单干都能完成 70%。但剩下的 30%——那些唯独来自另一份分析的新视角——才是这次改进真正的价值所在。
